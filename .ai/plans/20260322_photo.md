# Order Ticket System — Plan & Roadmap

**Created:** 2026-03-22  
**Status:** Phase 1 ✅ Complete | Phase 2 📋 Planned

---

## Vision

A lightweight ticket system anchored to orders, starting with proof-of-payment attachments and designed to grow into a full customer support channel.

The key insight: **tickets are separate from orders**. The Order stays clean. Tickets hang off it like attachments on an email.

---

## Phase 1 — Now: Payment Proof Attachments

**Goal:** Customer can attach up to 2 photos to their order as proof of payment (SINPE / bank transfer screenshot from their phone camera).

### What gets built
- `OrderTicket` model in Prisma (see schema below)
- Public API: `POST /api/orders/[orderNumber]/tickets` — create a ticket with attachments
- Public upload: `POST /api/upload/payment-proof` — returns a URL
- Order detail page (`/[locale]/orders/[orderNumber]`) — "Comprobante de pago" section:
  - Shows for SINPE / TRANSFER methods only, while order is unpaid
  - Mobile: triggers rear camera (`capture="environment"`)
  - Desktop: file browser + drag-and-drop
  - Max 2 photos, thumbnail previews, remove button
  - Uploads immediately on selection, attaches on confirm
- Admin order detail — shows submitted ticket attachments as thumbnails

### Schema (Phase 1)
```prisma
model OrderTicket {
  id          String       @id @default(cuid())
  order       Order        @relation(fields: [orderId], references: [id], onDelete: Cascade)
  orderId     String
  orderNumber String       // denormalized for easy public lookup
  type        TicketType   @default(PAYMENT_PROOF)
  attachments Json         @default("[]")  // String[] URLs, max 2
  message     String?      // optional customer note
  status      TicketStatus @default(OPEN)
  createdAt   DateTime     @default(now())
  updatedAt   DateTime     @updatedAt

  @@index([orderId])
  @@index([orderNumber])
  @@index([status])
}

enum TicketType   { PAYMENT_PROOF SUPPORT RETURN_REQUEST OTHER }
enum TicketStatus { OPEN IN_REVIEW RESOLVED CLOSED }
```

## Phase 2 — Near Future: Conversational Support Tickets

**Goal:** Customer can open a support ticket from any order. Admin can reply. Full message thread.

### New additions
- `TicketMessage` model — threaded messages (customer ↔ admin)
- Customer-side: "Abrir ticket de soporte" button on order detail page  
- Admin-side: dedicated ticket inbox in admin panel
- Ticket types: `SUPPORT`, `RETURN_REQUEST`, `EXCHANGE`, `OTHER`
- Status flow: `OPEN` → `IN_REVIEW` → `RESOLVED` / `CLOSED`
- Notifications: WhatsApp message to customer when admin replies
- Internal admin notes (invisible to customer)

### Schema additions
```prisma
model TicketMessage {
  id        String      @id @default(cuid())
  ticket    OrderTicket @relation(fields: [ticketId], references: [id], onDelete: Cascade)
  ticketId  String
  author    String      // 'customer' | 'admin'
  body      String
  attachments Json      @default("[]")  // String[] URLs
  createdAt DateTime    @default(now())

  @@index([ticketId])
}
```

## Phase 3 — Future: Full Support System

**Goal:** Decouple tickets from orders. Standalone support channel.

### Ideas
- Tickets without an order (general inquiries)
- Customer portal: view all tickets, status updates
- SLA / response time tracking
- Categories + tags for admin filtering
- Email + WhatsApp auto-notifications
- Admin assignment (which team member owns a ticket)
- Satisfaction rating after resolution

---

## Security Notes

- **Phase 1 API is public** — identified by `orderNumber` which only the customer knows (sent in checkout confirmation). No PII is exposed.
- All file uploads validated: type (JPG/PNG/WebP/HEIC), size (10MB max), random filenames.
- Phase 2+ should require customer login or a signed token (emailed link).
